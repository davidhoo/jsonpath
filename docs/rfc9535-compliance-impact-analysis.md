# RFC 9535 合规性改进影响分析

**分析日期：** 2026-05-06  
**分析依据：** `docs/rfc9535-compliance.md`  
**分析目的：** 评估完善项目 RFC 9535 合规性的影响  
**修订记录：** 2026-05-06 修正分类错误、类型定义、时间估算等问题

---

## 执行摘要

根据合规性文档，完善项目 RFC 9535 合规性需要：

| 维度 | 影响 |
|------|------|
| **API 兼容性** | 🔴 **大部分破坏性** - 行为修正和 API 变更需要主版本升级，少量新增语法可在次版本发布 |
| **架构** | 🔴 **重大重构** - 需要引入核心数据类型 |
| **工作量** | 🔴 **大** - 预计 7-10 周全职开发 |
| **风险** | 🔴 **高** - v3.0 升级时所有现有用户代码需要修改 |

**核心结论：** 大部分合规性改进是破坏性变更，无法在保持现有 API 的情况下实现 RFC 9535 合规。少数新增语法（存在性测试、`@` 引用）属于非破坏性新增，可先行实现。建议采用 v2.1.0 + v3.0.0 分阶段策略。

---

## 一、变更分类

### ⚠️ 重要说明

**大部分合规性改进是破坏性变更**，因为它们会改变现有查询的返回结果或行为。即使不改变函数签名，行为变化也会破坏依赖旧行为的用户代码。但存在性测试和 `@` 引用是纯新增语法，不影响任何现有查询，属于非破坏性新增（见 1.4 节）。

### 1.1 行为修正（改变现有查询结果）

这些变更不改变 API 签名，但会改变查询结果：

| 变更 | 为什么是破坏性 | 影响示例 |
|------|----------------|----------|
| 修复 `length()` 字符串计算 | `length("日本語")` 从 9 变为 3 | 依赖字节数的代码会得到错误结果 |
| 修复递归下降包含根节点 | `$..price` 可能多返回根节点的值 | 结果集变大，遍历逻辑可能出错 |
| 修复 `&&`/`\|\|` 优先级 | 混合表达式结果完全不同 | `a \|\| b && c` 语义变化 |
| 修复选择器错误处理 | 之前收到 error 的代码现在收到空结果 | 错误处理逻辑失效 |
| 修复过滤器评估所有元素 | 原始类型现在也被评估 | 结果集可能增大 |
| 添加字符串转义支持 | 转义字符现在被正确解析 | 依赖字面解析的代码会出错 |

### 1.2 API 签名变更（改变函数调用方式）

| 变更 | 当前 API | RFC 标准 API | 影响 |
|------|----------|--------------|------|
| `count()` 函数 | `count(array, value)` | `count(nodelist)` | 🔴 完全不同的签名 |
| `search()` 函数 | `search(array, pattern)` | `search(string, pattern)` | 🔴 完全不同的签名和返回类型 |
| `match()` 调用形式 | `@.field.match("pattern")` | `match(@.field, "pattern")` | 🔴 语法完全不同 |
| `value()` 函数 | 未实现 | `value(nodelist)` | 🟢 新增，不影响现有代码 |

### 1.3 返回类型变更（改变查询结果类型）

| 变更 | 当前 | RFC 标准 | 影响 |
|------|------|----------|------|
| Query 返回类型 | `interface{}` | `NodeList` | 🔴 所有调用代码需要修改 |
| 段返回类型 | `[]interface{}` | `NodeList` | 🔴 内部接口完全变化 |
| 错误处理 | 返回 `error` | 返回空 `NodeList` | 🔴 错误处理逻辑完全变化 |

### 1.4 非破坏性新增（不影响现有代码）

以下变更是纯新增语法，不改变任何现有查询的行为，可在次版本中发布：

| 变更 | 说明 | 工作量 |
|------|------|--------|
| 添加存在性测试 `[?@.name]` | 新增过滤器语法，测试字段是否存在 | 1-2 天 |
| 添加 `@` 单独引用 | 新增过滤器中 `@` 根节点引用语法 | 0.5-1 天 |

## 二、架构影响分析

### 2.1 需要引入的核心类型

根据 RFC 9535 Section 2.6 和合规性文档第七章，需要引入以下核心类型：

```go
// Node 表示 RFC 9535 节点（location + value）
type Node struct {
    Location string      // Normalized Path, e.g., $['store']['book'][0]
    Value    interface{}
}

// NodeList 表示节点列表
type NodeList []Node

// Nothing 表示 RFC 9535 的 Nothing 值（与 null 不同）
type Nothing struct{}

// LogicalType 表示逻辑值（三值逻辑）
type LogicalType int8
const (
    LogicalNothing LogicalType = iota  // 未定义/错误
    LogicalFalse                        // 逻辑假
    LogicalTrue                         // 逻辑真
)

// ValueType 表示 RFC 9535 的 ValueType
// 可以是 JSON 值或 Nothing
type ValueType interface{}
```

**影响：**
- 所有段的 `evaluate` 方法签名需要改变
- 所有函数的签名需要改变
- 查询结果类型需要改变
- 需要实现 NodeList 的序列化/反序列化
- 需要实现 Normalized Path 生成

### 2.2 需要修改的核心组件

| 组件 | 当前签名 | 新签名 | 影响范围 |
|------|----------|--------|----------|
| `segment.evaluate` | `(value interface{}) ([]interface{}, error)` | `(node Node) (NodeList, error)` | 所有段实现 |
| `Query` | `(data interface{}, path string) (interface{}, error)` | `(data interface{}, path string) (NodeList, error)` | 公共 API |

### 2.3 函数接口需要分别定义

RFC 9535 的函数有不同的签名，不能用统一接口：

```go
// LengthFunc 签名
type LengthFunc func(value ValueType) ValueType

// CountFunc 签名
type CountFunc func(nodes NodeList) ValueType

// MatchFunc 签名
type MatchFunc func(str ValueType, pattern ValueType) LogicalType

// SearchFunc 签名
type SearchFunc func(str ValueType, pattern ValueType) LogicalType

// ValueFunc 签名
type ValueFunc func(nodes NodeList) ValueType
```

### 2.4 新增功能模块

| 模块 | 说明 | 工作量 |
|------|------|--------|
| Normalized Path 生成器 | 为每个节点生成唯一路径标识 | 3-5 天 |
| Nothing 类型处理 | 贯穿整个系统的 Nothing 语义 | 2-3 天 |
| I-Regexp 解析器 | 实现 RFC 9485 正则表达式子集 | 5-8 天 |
| 类型系统 | 实现 ValueType/LogicalType/NodesType 区分 | 2-3 天 |

## 七、兼容性策略

### 7.1 策略 A：大爆炸式升级（推荐）

**方法：** 直接发布 v3.0.0，包含所有 RFC 9535 合规变更

**优点：**
- 一次性解决所有问题
- 代码库干净
- 无需维护兼容层

**缺点：**
- 用户需要一次性迁移所有代码
- 可能导致用户流失

**迁移指南示例：**
```markdown
# 从 v2.x 迁移到 v3.0

## Breaking Changes

### 1. Query 返回类型变化
```go
// v2.x
result, err := jsonpath.Query(data, "$.store.book")
// result 类型: interface{}

// v3.0
result, err := jsonpath.Query(data, "$.store.book")
// result 类型: NodeList
// 获取值: result[0].Value
```

### 2. count() 函数重命名
```go
// v2.x
"$.items.count('value')"  // 计数特定值

// v3.0
"$.items.occurrences('value')"  // 计数特定值（非标准扩展）
"?count(@.items[*]) > 0"  // RFC 标准：计算节点数
```

### 3. search() 函数重命名
```go
// v2.x
"$.items.search('pattern')"  // 过滤数组

// v3.0
"$.items.filterMatch('pattern')"  // 过滤数组（非标准扩展）
"?search(@.name, 'pattern')"  // RFC 标准：字符串子串搜索
```

### 4. match() 语法变化
```go
// v2.x
"?@.name.match('S')"     // 方法式调用，部分匹配：匹配任何包含 "S" 的字符串（如 "AliceSmith"）

// v3.0
"?match(@.name, 'S')"    // 标准函数调用，全串匹配：只匹配恰好为 "S" 的字符串
```
```

### 7.2 策略 B：渐进式迁移

**方法：** 发布 v2.x 兼容层，然后 v3.0 移除

**步骤：**
1. v2.5: 添加新 API，旧 API 标记为 deprecated
2. v2.6: 添加迁移工具
3. v3.0: 移除旧 API

**优点：**
- 用户有时间迁移
- 可以逐步适应

**缺点：**
- 需要维护两套 API
- 代码库复杂
- 开发周期长（可能 3-4 个月）

### 7.3 策略 C：并行实现

**方法：** 创建 `jsonpath/rfc9535` 子包

```go
import "github.com/davidhoo/jsonpath/rfc9535"

// RFC 9535 完全兼容版本
result, err := rfc9535.Query(data, "$.store.book")

// 旧版本保持不变
result, err := jsonpath.Query(data, "$.store.book")
```

**优点：**
- 完全向后兼容
- 用户可以选择迁移时机

**缺点：**
- 代码重复
- 维护成本高
- 可能导致社区分裂

### 7.4 策略 D：v2.1.0 中间路径（推荐先执行）

**方法：** 先发布 v2.1.0，只做行为修正和新增语法，不改 API 签名。然后再决定是否推进 v3.0。

**v2.1.0 范围：**

| 变更 | 性质 | 理由 |
|------|------|------|
| 修复 `length()` 字符串计算 | bug fix | 当前实现不符合 JSON 语义 |
| 修复递归下降包含根节点 | bug fix | 当前实现不符合 RFC 规范 |
| 修复 `&&`/`\|\|` 优先级 | bug fix | 当前实现不符合标准优先级 |
| 修复选择器错误处理 | bug fix | 当前实现行为不正确 |
| 添加存在性测试 `[?@.name]` | 新增语法 | 纯新增，不影响现有代码 |
| 添加 `@` 单独引用 | 新增语法 | 纯新增，不影响现有代码 |

**优点：**
- 以"bug fix"名义在次版本发布，语义版本规范允许
- 用户影响可控，不需要修改任何调用代码
- 立即提升 RFC 合规性，为后续升级铺路
- 可以先发布 v2.1.0，再根据用户反馈决定是否推进 v3.0

**缺点：**
- 不解决 API 签名问题（`count()`、`search()`、`match()` 调用形式）
- 不引入 NodeList/Node 类型
- 仍需要 v3.0 才能完全合规

**工作量：约 2-3 周**

**迁移示例：**
```go
// v2.0.x → v2.1.0：无需修改任何代码
// 以下行为会自动修正：
// - length("日本語") 从 9 变为 3（正确）
// - $..price 不再多返回根节点（正确）
// - a || b && c 按标准优先级解析（正确）
// - 选择器错误返回空结果而非 error（正确）
// 新增可用语法：
// - [?@.name] 存在性测试
// - [?@ > 5] @ 单独引用
```

## 八、建议

### 8.1 推荐策略

**推荐分两阶段推进：先执行策略 D（v2.1.0），再根据反馈决定是否推进策略 A（v3.0.0）。**

**第一阶段：策略 D（v2.1.0，2-3 周）**

1. **风险最低**：行为修正以 bug fix 名义发布，不破坏 API
2. **立即见效**：6 项合规改进可在次版本落地
3. **收集反馈**：通过 v2.1.0 用户反馈评估 v3.0 的必要性和用户接受度

**第二阶段：策略 A（v3.0.0，约 8-11 周）**

如果 v2.1.0 反馈积极且用户需要完全合规，再推进 v3.0.0：
1. **代码库规模适中**：当前代码库不大，可以一次性重构
2. **用户群体可控**：可以通过 GitHub 通知用户
3. **维护成本低**：不需要长期维护两套 API

### 8.2 实施步骤

**第一阶段：v2.1.0（第 1-3 周）**

1. **第 1 周**
   - 获取并集成 RFC 9535 官方测试套件（324 个用例）
   - 运行测试，建立当前通过率基线
   - 修复 `length()` 字符串计算
   - 修复 `&&`/`||` 优先级

2. **第 2 周**
   - 修复递归下降包含根节点
   - 修复选择器错误处理
   - 添加存在性测试 `[?@.name]`
   - 添加 `@` 单独引用

3. **第 3 周**
   - 运行完整测试套件，验证通过率提升
   - 编写 v2.1.0 变更日志
   - 发布 v2.1.0

**第二阶段：v3.0.0（如需要，第 4-13 周）**

1. **准备阶段（第 4 周前半）**

## 十、结论

完善 RFC 9535 合规性是一个**重大工程**，需要：

1. **主版本升级**（v2.x → v3.0.0）
2. **架构重构**（引入 Node/NodeList/Nothing/LogicalType 类型）
3. **大部分为破坏性 API 变更**（行为修正破坏旧行为；存在性测试和 @ 引用为非破坏性新增）
4. **约 8-11 周的全职开发周期**（如先执行 v2.1.0 则需额外 2-3 周）
5. **通过 324 个 RFC 官方测试用例**
